## What Went Well

1. **Incremental architecture** — Each story built cleanly on the previous. The `editorStore` evolved from simple CRUD (2.1) → field timestamps (2.4) → write-lock (2.5) → temporal undo (2.6) without breaking changes.

2. **Field-level LWW protocol** — The per-field conflict resolution (Story 2.4) is a solid foundation. Server-as-clock-authority prevents client clock skew issues. The protocol is convergent and well-tested (10 backend + 17 frontend unit tests).

3. **Code review process caught real bugs** — Every round surfaced genuine issues (not just style nits). Examples: IntersectionObserver partial-delta bug (2.3), clock-skew infinite loop (2.4), SSE line buffering data loss (2.5), zundo partialize missing (2.6).

4. **Velocity improved over the epic** — Story 2.2 needed 24 patches. Story 2.6 needed only 3. The team learned from each round.

5. **Spec quality improved** — Story 2.4 and 2.5 specs went through self-review passes that caught critical issues (fake-indexeddb missing, PII service vs interceptor confusion) BEFORE implementation.

## What Didn't Go Well (Juragan's Key Concern)

### Pattern: Multi-round review cycles with persistent findings

**Data:**
| Story | Round 1 Patches | Round 2 Patches | Total Rounds |
|-------|----------------|----------------|--------------|
| 2.2 | 24 | — | 1 |
| 2.3 | 14 | 7 | 3 |
| 2.4 | 5 | 1 | 2 |
| 2.5 | 6 | 3 | 2 |
| 2.6 | 3 | — | 1 |

**Root causes identified:**

1. **First-pass code generation lacks defensive programming** — Guards for null, edge cases, browser compat (older Safari, missing APIs) are consistently missed on first pass. They're caught in review but shouldn't need to be.

2. **SSR/hydration edge cases not considered upfront** — `useBreakpoint` SSR safety, `useReducedMotion` first-paint flash, `Set<T>` iteration with es5 target — all discovered in review, not during implementation.

3. **Review findings from round 1 sometimes introduce NEW bugs in round 2** — The fix for one issue creates another (e.g., fixing IO partial-delta introduced the clock-skew loop in 2.4; fixing the controller try/catch in 2.5 was attempted but strReplace failed silently).

4. **Integration tests are sparse** — Unit tests cover logic well (47 tests), but RTL/MSW integration tests that verify the full hook→store→API flow are consistently deferred. This means bugs at the integration boundary survive until manual smoke testing.

---

## Key Insights

1. **Spec self-review is high-ROI** — Stories 2.4 and 2.5 had spec reviews that caught 5+ critical issues each BEFORE any code was written. This prevented entire classes of bugs (wrong dependency assumptions, incorrect API patterns, missing privacy requirements).

2. **The "fix introduces new bug" pattern is the most expensive** — Round 2 findings are often caused by round 1 fixes. A more careful approach to patches (read surrounding code, consider side effects) would reduce this.

3. **Architecture doc drift is real** — The 150ms vs 800ms IDB debounce discrepancy (architecture says one thing, AC says another) caused confusion. The architecture doc needs reconciliation.

4. **Bundle budget is healthy** — 202 kB FirstLoad is well within the 180 kB initial JS budget (NFR7). The editor route at 30.4 kB is reasonable for the feature density.

5. **Technical debt is manageable** — 10 items in deferred-work.md, mostly auth-flow related (Story 1.4 territory) and performance optimizations. None are blocking Epic 3.
